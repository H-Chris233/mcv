# Crypto Design

Status: v0.1 MVP implemented in Rust.

The Rust core uses Shamir Secret Sharing over GF(256), Argon2id, HKDF-SHA256, XChaCha20-Poly1305, fixed-order CBOR serialization, and caller-provided CSPRNG randomness.

## Secrets

- `master_secret`: random 32-byte Vault secret generated by Rust.
- `shamir_share`: one serialized GF(256) share of `master_secret`.
- `data_fragment`: one serialized GF(256) share of the encoded Vault Blob.
- `password_key`: Argon2id output for Vault unlock.
- `card_wrap_key`: Argon2id output for one card payload.
- `final_key`: 32-byte Vault data encryption key derived by HKDF-SHA256.

## Key Derivation

`password_key = Argon2id(user_password, vault_salt, vault_kdf_params)`

`final_key = HKDF-SHA256(master_secret || password_key, vault_salt, "mcv-vault-v1")`

Each card share is separately wrapped with:

`card_wrap_key = Argon2id(user_password, card_salt, card_kdf_params)`

## Create Flow

1. Rust validates threshold, total, and plaintext length.
2. Rust generates `vault_id`, `scheme_id`, and `master_secret`.
3. Rust splits `master_secret` into `total` Shamir Shares with threshold `threshold`.
4. Rust derives `password_key` and `final_key`.
5. Rust encrypts Vault Plaintext v1 into Vault Blob v1.
6. Rust splits the encoded Vault Blob into `total` Data Fragments with threshold `threshold`.
7. Rust encrypts each Share with XChaCha20-Poly1305 using a per-card `card_wrap_key`, `card_salt`, and `card_nonce`.
8. Android stores only non-recovery metadata and writes each Card Payload to a CUID Card.

## Unlock Flow

1. Rust decodes card payloads and validates magic, version, `vault_id`, `scheme_id`, algorithm IDs, threshold, total, and `share_index`.
2. Rust rejects duplicate `share_index` values.
3. Rust recovers the encoded Vault Blob from threshold Data Fragments and validates magic, version, IDs, algorithm IDs, threshold, total, salt, nonce, and ciphertext shape.
4. Each encrypted Share is decrypted with the password-derived card key and Card Payload header AAD.
5. Once enough valid Shares are available, Rust recovers `master_secret`.
6. Rust derives `final_key` from `master_secret` and `password_key`.
7. Rust decrypts Vault Blob ciphertext using Vault Blob header AAD.

## Update Flow

After unlock, Android passes the new serialized Vault Plaintext v1 to Rust. Rust verifies the old card set, generates a fresh `scheme_id`, `master_secret`, `vault_salt`, and `vault_nonce`, then returns a replacement Card Payload set. The UI must rewrite all cards before treating the update as complete.

## AEAD AAD

Card Payload AAD is the fixed-order header without `encrypted_share`. Vault Blob AAD is the fixed-order header without `ciphertext`. A header mutation must make decryption fail.

## Nonces

XChaCha20-Poly1305 nonces are 24 bytes. Every card share encryption and Vault Blob encryption uses a fresh nonce from the caller-provided CSPRNG. Nonces are stored in the corresponding payload and authenticated through AAD.

## Shamir Implementation

`mcv-shamir` wraps `blahaj` behind the local `SecretSharing` trait. Upper layers do not depend on `blahaj` directly, so the implementation can be replaced without changing Card Payload or Vault Blob orchestration code.

## Argon2id Defaults

Production default Argon2id params are:

```text
memory_cost_kib = 32768
time_cost = 3
parallelism = 1
```

Tests and deterministic vectors use lower explicit params so CI remains fast. These test params are not production defaults.

## Error Surface

Rust keeps distinct errors for malformed payloads, wrong vault ID, wrong scheme ID, duplicate shares, card AEAD failure, and vault AEAD failure. UI code should still avoid exposing unnecessary cryptographic detail to ordinary users.

## Memory Handling

Rust uses `zeroize`/`secrecy` boundaries for derived key material where practical. The MVP still does not claim protection against a compromised OS, live memory extraction after unlock, or side-channel attacks.
